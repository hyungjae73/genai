## 1. 監視・クロール

### 1.1 定期クロール

| # | 要件 |
|---|------|
| 1 | 全登録監視サイトを日次スケジュールで自動クロールする |
| 2 | 1サイトあたり5分以内にクロールを完了する |
| 3 | 同一ドメインへのリクエスト間隔は最低10秒を確保する（レートリミット） |
| 4 | robots.txt の指示を遵守する |
| 5 | クロール失敗時は指数バックオフで最大3回リトライする |
| 6 | クロール結果（HTML・タイムスタンプ・URL・ステータスコード）をDBに保存する |

### 1.2 手動クロール

| # | 要件 |
|---|------|
| 7 | 管理者はサイト一覧・サイト管理画面から任意のサイトを即時クロールできる |
| 8 | クロール実行中はスピナーを表示し、ボタンを無効化する |
| 9 | クロール完了時に最終クロール日時を更新し、差異が検出された場合はアラートを自動生成する |
| 10 | 同一サイトへの重複クロール実行を防止する（409 Conflict） |
| 11 | クロール完了・失敗時にトースト通知を表示する |

### 1.3 クロール信頼性（ボット対策）

| # | 要件 |
|---|------|
| 12 | ブラウザのロケールと Accept-Language ヘッダーを日本語（ja-JP）に設定してクロールする |
| 13 | ページ上のモーダル・Cookie同意バナーを自動的に閉じてからスクリーンショットを取得する |
| 14 | サイトごとにカスタムPlaywrightアクション（クリック・待機・入力等）を登録・実行できる |
| 15 | HTTP 200 を返すが実際はボット対策ページ（CAPTCHA等）である Soft Block を検知する |
| 16 | ボット対策が強いサイトは外部スクレイピングサービス（SaaS）経由でクロールできる |
| 17 | サイトのクロール成功率が閾値（デフォルト80%）を下回った場合、自動的に最適なフェッチ戦略に切り替える |

### 1.4 クロールスケジュール管理

| # | 要件 |
|---|------|
| 18 | サイトごとにクロール優先度（高/通常/低）とクロール間隔（分単位）を設定できる |
| 19 | 次回クロール予定日時を表示し、「今すぐ実行」ボタンで即時クロールをトリガーできる |
| 20 | 変更のないサイトはフルクロールをスキップし、クロール負荷を削減する（デルタクロール） |

## 2. 決済情報抽出・検証

### 2.1 情報抽出

| # | 要件 |
|---|------|
| 1 | HTMLから価格・決済方法・手数料・サブスクリプション条件を自動抽出する |
| 2 | JSON-LD・Microdata・Open Graph 等の構造化データから全バリアント価格を優先抽出する |
| 3 | Shopify サイトは product.json API から全バリアント価格を取得する |
| 4 | 構造化データが取得できない場合は HTML 解析にフォールバックする |
| 5 | スクリーンショットから OCR でテキストを抽出し、視覚的な価格・条件情報を補完する |
| 6 | 抽出データに信頼度スコア（0.0〜1.0）を付与する |
| 7 | 抽出結果をDBに永続化し、後から参照・分析できるようにする |

### 2.2 契約条件との照合

| # | 要件 |
|---|------|
| 8 | 抽出した価格・決済方法・手数料・サブスク条件を登録済み契約条件と照合する |
| 9 | 全バリアント価格を個別に契約条件と比較し、いずれかが違反している場合に検出する |
| 10 | 違反検出時は違反種別・期待値・実際の値・データソースを記録する |
| 11 | HTML抽出データと OCR 抽出データの両方を契約条件と照合し、各データソースの違反を個別に記録する |

### 2.3 証拠保全

| # | 要件 |
|---|------|
| 12 | スクリーンショットから価格表示・注意書き・定期購入条件の領域（ROI）を切り出して保存する |
| 13 | 各証拠データに証拠種別（価格表示・条件通知・定期購入条件・一般）を分類して付与する |
| 14 | 証拠データは検証結果と紐付けて管理し、後から参照・検索できる |

### 2.4 抽出結果レビュー

| # | 要件 |
|---|------|
| 15 | レビュアーはスクリーンショットと抽出データを並べて確認できる |
| 16 | 抽出ミスを発見した場合、その場でデータを手動修正できる |
| 17 | 修正内容は変更履歴として記録される（誰がいつ何を変更したか） |
| 18 | レビュアーは抽出結果を「承認」または「却下（再クロール要求）」できる |
| 19 | HTML抽出とOCR抽出の結果を並べて比較できる |
| 20 | 両方の抽出が失敗した場合、スクリーンショットと生HTMLを表示して目視確認できる |

## 3. 偽サイト検知

### 3.1 検知フロー

| # | 要件 |
|---|------|
| 1 | 登録済み監視サイトのドメインに類似するドメインを定期的にスキャンする |
| 2 | ドメイン類似度（Damerau-Levenshtein距離＋ビジュアル類似文字正規化）を算出する |
| 3 | 類似度80%以上のドメインを偽サイト候補としてフラグを立てる |
| 4 | 候補ドメインのコンテンツを取得し、正規サイトとのコンテンツ類似度を算出する |
| 5 | コンテンツ類似度70%以上で偽サイトとして確定し、critical アラートを生成する |
| 6 | コンテンツ類似度はテキスト（重み0.4）・重要フィールド（0.3）・構造（0.15）・視覚（0.15）の加重平均で算出する |

### 3.2 アラート表示

| # | 要件 |
|---|------|
| 7 | 偽サイトアラートには検知ドメイン・正規ドメイン・類似度スコア・検知日時を表示する |
| 8 | 未解決の偽サイトアラートには「TakeDown対応が必要」の警告バナーを表示する |
| 9 | アラート一覧で契約違反アラートと偽サイトアラートを種別フィルターで絞り込める |
| 10 | ダッシュボードに偽サイト検知数と未解決件数を統計カードとして表示する |

### 3.3 偽サイト候補確認UI

| # | 要件 |
|---|------|
| 11 | サイト詳細画面でダークサイト候補一覧（ドメイン・マッチ種別・類似度・リスクスコア）を確認できる |
| 12 | 候補の「コンテンツ確認」から正規サイトと候補サイトのテキスト・画像を並列比較できる |
| 13 | 契約条件との乖離リスト（フィールド名・正規値・候補値・重要度）を表示する |

## 4. ダークパターン検出

### 4.1 CSS・視覚的欺瞞検出

| # | 要件 |
|---|------|
| 1 | ページ上のテキスト要素のコントラスト比が2.0:1未満の場合、低コントラスト欺瞞として検出する |
| 2 | 条件テキストのフォントサイズが主要価格表示の25%未満の場合、フォントサイズ異常として検出する |
| 3 | オフスクリーン配置・opacity:0・font-size:0・overflow:hidden によるテキスト隠蔽を検出する |

### 4.2 LLMセマンティック分類

| # | 要件 |
|---|------|
| 4 | ページテキストを外部LLM APIに送信し、隠れた定期購入条件・ダークパターンを分類する |
| 5 | LLM分類結果の確信度が0.7以上かつ定期購入条件ありの場合、違反として記録する |
| 6 | コンプライアンス担当者がDBに自然言語プロンプトを登録するだけで新しい検出ルールを追加できる（LLM as a Judge） |

### 4.3 ユーザージャーニー検出

| # | 要件 |
|---|------|
| 7 | カート追加・チェックアウト遷移時に動的に出現する隠し料金・アップセルモーダルを検出する |
| 8 | サイトごとにジャーニースクリプト（カート追加→チェックアウト等の操作手順）を登録できる |
| 9 | ジャーニー実行前後で可視要素を比較し、新規出現した料金要素・モーダルを検出する |

### 4.4 UI/UXトラップ検出

| # | 要件 |
|---|------|
| 10 | デフォルトでチェック済みの有料オプション・定期購入チェックボックスを検出する（sneak_into_basket） |
| 11 | デフォルト選択された定期購入ラジオボタンを検出する（default_subscription） |
| 12 | 解約条件テキストが選択要素から20要素以上離れている場合を検出する（distant_cancellation_terms） |
| 13 | 「いいえ、節約したくありません」等のコンファームシェイミングパターンを検出する |

### 4.5 総合スコアと分類体系

| # | 要件 |
|---|------|
| 14 | 4つの検出アプローチの結果をMax Poolingで統合した総合ダークパターンスコア（0〜1）を算出する |
| 15 | スコアが閾値（デフォルト0.6）以上の場合、high_dark_pattern_risk 違反を生成する |
| 16 | 検出結果は標準化された分類体系（visual_deception / hidden_subscription / sneak_into_basket / default_subscription / confirmshaming / distant_cancellation_terms / hidden_fees / urgency_pattern / price_manipulation / misleading_ui / misleading_font_size）で記録する |

### 4.6 動的検出ルール管理

| # | 要件 |
|---|------|
| 17 | 管理者はLLMベースの動的検出ルールを管理画面から作成・編集・削除できる |
| 18 | ルールにはルール名・説明・重要度・確信度閾値・プロンプトテンプレートを設定できる |
| 19 | プロンプトテンプレートには `{page_text}` プレースホルダーが必須（登録時にバリデーション） |
| 20 | ルールはグローバル適用またはサイト単位での適用範囲を設定できる |

## 5. アラート・通知

### 5.1 アラート生成

| # | 要件 |
|---|------|
| 1 | 契約条件違反・偽サイト確定・ダークパターン高リスク検出時にアラートを自動生成する |
| 2 | アラートには違反種別・重要度（critical/high/medium/low）・サイト名・検出日時を記録する |
| 3 | 偽サイトアラートには検知ドメイン・正規ドメイン・ドメイン類似度・コンテンツ類似度を記録する |

### 5.2 Slack通知

| # | 要件 |
|---|------|
| 4 | 違反検出時にSlack Incoming Webhook へ通知を送信する |
| 5 | 通知メッセージにはサイト名・違反種別・検出価格・証拠スクリーンショットURL・重要度・検出日時を含める |
| 6 | 重要度に応じてメッセージを色分けする（warning: 黄、critical: 赤、info: 青） |
| 7 | 送信失敗時は指数バックオフで最大3回リトライする |

### 5.3 メール通知

| # | 要件 |
|---|------|
| 8 | 違反検出時に監視サイトに紐づく顧客のメールアドレスへ通知メールを送信する |
| 9 | 件名は「[決済条件監視] {重要度}: {サイト名} でダークパターン違反を検出」の形式とする |
| 10 | 送信失敗時は指数バックオフで最大3回リトライする |

### 5.4 重複通知抑制

| # | 要件 |
|---|------|
| 11 | 同一サイト・同一違反種別の通知は24時間以内に重複送信しない（抑制ウィンドウは設定可能） |
| 12 | 同一サイトで異なる違反種別が検出された場合は各違反種別を独立して判定する |

### 5.5 通知設定管理

| # | 要件 |
|---|------|
| 13 | 管理者はサイト詳細画面からSlack/メール通知の有効/無効・送信先・抑制ウィンドウを設定できる |
| 14 | Slack Webhook URL はマスク表示する（末尾8文字以外を `***` で表示） |
| 15 | 通知送信履歴（チャネル・ステータス・送信日時）を確認できる |

## 6. 手動審査ワークフロー

### 6.1 審査キュー

| # | 要件 |
|---|------|
| 1 | 重要度 critical/high/medium のアラートは自動的に審査キューに投入される |
| 2 | ダークパターンスコアが0.7以上の場合、審査キューに投入される |
| 3 | 偽サイトアラートは priority: critical で審査キューに投入される |
| 4 | 同一アラートへの重複投入は防止する |

### 6.2 一次審査（reviewer）

| # | 要件 |
|---|------|
| 5 | reviewer は審査キューから案件を取得し、関連するアラート・違反・ダークパターン・偽サイト情報を一画面で確認できる |
| 6 | reviewer は案件を自分に割り当て（in_review）、判定（承認/却下/エスカレーション）できる |
| 7 | 判定時はコメント（理由）の入力が必須 |
| 8 | 承認（問題なし）→ アラートを解決済みに更新 |
| 9 | 却下（違反確定）→ アラートを未解決のまま維持し、顧客へ違反確定通知を送信 |
| 10 | エスカレーション → 案件を二次審査キューに移し、admin へ通知 |

### 6.3 二次審査（admin）

| # | 要件 |
|---|------|
| 11 | admin はエスカレーション案件の一覧を確認し、最終承認または最終却下を行える |
| 12 | reviewer は二次審査判定を行えない（403） |

### 6.4 審査記録・状態管理

| # | 要件 |
|---|------|
| 13 | 全審査判定は審査者・判定結果・コメント・審査段階（一次/二次）とともに記録される |
| 14 | 状態遷移は pending → in_review → approved/rejected/escalated → approved/rejected の順序のみ許可する |
| 15 | 最終状態（approved/rejected）からの遷移は不可 |

### 6.5 審査ダッシュボード

| # | 要件 |
|---|------|
| 16 | 未審査・審査中・エスカレーション・承認済み・却下済みの件数を表示する |
| 17 | 優先度別・審査種別別の未審査件数を表示する |
| 18 | 審査キューを優先度・ステータス・種別・担当者でフィルタリングできる |

## 7. 契約条件管理

### 7.1 基本CRUD

| # | 要件 |
|---|------|
| 1 | 管理者はサイトごとに契約条件（価格・決済方法・手数料・サブスク条件）を登録・更新・削除できる |
| 2 | 契約条件の更新時は新バージョンを作成し、変更履歴を保持する |
| 3 | 削除はソフトデリートとし、監査目的でデータを保持する |
| 4 | 契約条件の更新時は対象サイトの即時再検証をトリガーする |

### 7.2 カテゴリ管理

| # | 要件 |
|---|------|
| 5 | 管理者は商品・サービスカテゴリを自由に追加・編集・削除できる |
| 6 | カテゴリにはカラー・説明を設定できる |
| 7 | カテゴリ削除時は関連サイト・契約条件を「未分類」に移動する |
| 8 | 監視サイト・契約条件をカテゴリで分類・フィルタリングできる |

### 7.3 動的フィールド（カテゴリ別追加項目）

| # | 要件 |
|---|------|
| 9 | 管理者はカテゴリごとに追加フィールド（フィールド名・型・必須/任意・バリデーションルール・表示順）を定義できる |
| 10 | フィールド型はテキスト・数値・通貨・パーセンテージ・日付・真偽値・リストをサポートする |
| 11 | 契約条件登録フォームでカテゴリを選択すると、そのカテゴリの追加フィールドが自動表示される |
| 12 | 追加フィールドの入力値はバリデーションルール（min/max/pattern/options）に従ってリアルタイム検証される |
| 13 | 追加フィールドの値は契約条件と一緒に保存・表示・編集できる |

## 8. ユーザ管理・権限管理

### 8.1 ユーザ認証

| # | 要件 |
|---|------|
| 1 | ユーザ名とパスワードでログインし、JWTアクセストークン（30分）とリフレッシュトークン（7日）を発行する |
| 2 | アクセストークン期限切れ時はリフレッシュトークンで自動更新する |
| 3 | ログアウト時はトークンを無効化する |
| 4 | 無効化されたアカウントはログインを拒否する |
| 5 | ログイン試行は5分間に10回までに制限する（レートリミット） |

### 8.2 ロールベースアクセス制御

| # | 要件 |
|---|------|
| 6 | **admin**: 全機能にアクセス可能（ユーザ管理・ルール管理・サイト管理・審査・設定変更） |
| 7 | **reviewer**: サイト閲覧・アラート確認・手動審査（承認/却下/エスカレーション）が可能 |
| 8 | **viewer**: 読み取り専用（ダッシュボード・アラート閲覧・結果参照） |
| 9 | 権限のないエンドポイントへのアクセスは 403 を返す |
| 10 | viewer ロールのユーザには書き込み操作のUIボタン（作成・編集・削除）を非表示にする |

### 8.3 ユーザ管理

| # | 要件 |
|---|------|
| 11 | admin はユーザの作成・一覧・詳細・更新・無効化ができる |
| 12 | ユーザ名・メールアドレスは一意 |
| 13 | 自分自身のアカウントは無効化できない |
| 14 | パスワードは8文字以上・英大文字・英小文字・数字を各1文字以上含む必要がある |
| 15 | 初回デプロイ時に環境変数から初期 admin アカウントを自動作成する |

### 8.4 監査ログ

| # | 要件 |
|---|------|
| 16 | 全書き込み操作は実行ユーザ・アクション・対象リソースとともに監査ログに記録される |
| 17 | ログイン成功・失敗も監査ログに記録される |
| 18 | admin は監査ログをユーザ名・アクション・リソース・日時範囲でフィルタリングして閲覧できる |

## 9. 管理画面・ダッシュボード

### 9.1 ダッシュボード

| # | 要件 |
|---|------|
| 1 | 監視サイト数・違反数・成功率・偽サイト検知数・高リスクサイト数をカード表示する |
| 2 | ダークパターンスコア分布（低/中/高リスク）とカテゴリ別検出数をグラフ表示する |
| 3 | データは30秒ごとに自動更新する |

### 9.2 サイト管理（階層型ビュー）

| # | 要件 |
|---|------|
| 4 | 顧客→サイトの階層構造で監視サイトを一覧表示する |
| 5 | サイト行を展開すると契約条件・スクリーンショット・検証比較・アラート・ダークパターン・スケジュールの各タブが表示される |
| 6 | 顧客名・サイト名・カテゴリ・ステータスでフィルタリングできる |

### 9.3 アラート一覧
